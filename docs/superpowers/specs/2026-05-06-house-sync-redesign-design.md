# House Sync Redesign

**Date:** 2026-05-06
**Status:** Draft — design approved, planning next
**Scope:** `HouseSyncService`, `HouseRepository`, `House` entity + Liquibase schema, env-var cleanup

## Problem

The current `HouseSyncService` works at today's data volume but has four real defects that will get worse over time or already produce wrong data:

1. **Queue points can be wiped by null.** `applyDetailToHouse` calls `house.setQueuePoints(detail.queuePointsCurrentPositionX())` unconditionally. The API returns null when the cut-off has not been established; if a later fetch happens to return null again (or the field is intermittently absent), a previously captured value is overwritten with null.
2. **`syncHouseList` reconcile loads every row ever inserted.** `houseRepository.findAll()` returns ended rows too. The active-set diff doesn't need them. Memory and reconcile time grow linearly with historical volume.
3. **A re-listed property silently mutates the old row.** When an external `id` reappears in the API after the listing ended, line 66 (`house.setEndDate(null)`) re-activates the same row. History (queue points, prior `endDate`) is lost. There is no concept of a listing *instance*.
4. **Detail-fetch cadence is an arbitrary 60s tick.** With even a few dozen active houses, most ticks query the DB and find nothing, and `DETAIL_API_CALL_INTERVAL_SECONDS` / `DETAIL_SYNC_BATCH_SIZE` are magic numbers unrelated to actual workload. Ended-but-still-listed houses (deadline passed, API hasn't dropped them yet) keep getting fetched until they disappear from the API.

## Goals

- Queue points, once captured, never disappear due to a null API response.
- `(id, availableFrom)` uniquely identifies a listing instance. Same external id with a different `availableFrom` is a new listing.
- `syncHouseList` only ever loads active rows; ended rows are immutable history.
- Detail-fetch cadence is a *consequence* of how many active houses exist and the chosen refresh window, not a hand-tuned constant.
- Houses whose `applicationDeadline` has passed are ended proactively and stop consuming detail-fetch budget.
- Dead env vars are removed.

## Non-goals

- Refactoring the legacy `Homes` / `HomesRepository` / `legacy/*` surface. Those stay as-is.
- Reintroducing or redesigning Slack notifications. `SlackService` / `SlackClient` remain untouched.
- Changing the list / detail HTTP client (`VaxjobostaderClient`).
- Backfilling historical `availableFrom` values for any rows that currently lack one (see migration note below).

## Design

### 1. Sticky queue points

`applyDetailToHouse` updates `queuePoints` only when the incoming value is non-null:

```java
Integer incoming = detail.queuePointsCurrentPositionX();
if (incoming != null) {
    house.setQueuePoints(incoming);
}
```

Rationale: the field name is "current position X" but a null response empirically means "API hasn't computed it for this listing yet," not "the cut-off was reset to nothing." Latest non-null wins. We accept that if the API legitimately re-publishes a different cut-off later, that newer non-null value will overwrite the older one — which is the desired behaviour.

### 2. Identity model: `(id, availableFrom)`

A listing instance is uniquely the pair `(id, availableFrom)`. Same external id with a different `availableFrom` is a different listing.

**Schema change (Liquibase):**

- Add an autogenerated surrogate primary key (e.g. `internal_id BIGINT GENERATED BY DEFAULT AS IDENTITY`). Exact column name and identity syntax decided during planning to match existing migration conventions and the target DB engine.
- Drop the existing PK on `id`; replace with a unique constraint on `(id, available_from)`.
- Index `(id)` (now non-unique) for the detail-sync lookup path.
- The existing `findHousesNeedingDetailRefresh` JPQL is unchanged — it filters on `endDate IS NULL` and `lastDetailFetchedAt`, neither of which depends on the PK shape.

**Why surrogate over composite `@IdClass` / `@EmbeddedId`:** simpler JPA mapping, simpler `findById(Long)` semantics, no awkwardness when other code paths just want "the active row for this external id." The composite is enforced by the unique constraint, not by the PK shape.

**Null `availableFrom` policy:** a null `available_from` cannot participate in a unique constraint, so a row with null would break the model. The implementation plan must decide one of:
- reject API items missing `availableFrom` (log + skip),
- treat missing `availableFrom` as a sentinel value (e.g. `epoch`),
- require a non-null DB constraint and fail loudly if violated.

Recommended default: **reject + log**, on the basis that we have not observed null `availableFrom` in production data and silent sentinels make later debugging harder.

**Existing data:** the current table uses `id` as PK with no `available_from` uniqueness. Migration adds the surrogate PK; existing rows keep their `available_from` values. If any existing rows have null `available_from`, the migration must either backfill them or fail — to be decided in planning.

### 3. `syncHouseList` reconcile

```
load activeRows = houseRepository.findAllByEndDateIsNull()
build activeKey = Map<(id, availableFrom), House> from activeRows
fetch apiItems

for each apiItem:
    key = (apiItem.id, apiItem.availableFrom)
    if key in activeKey:
        update list-level fields on the matched row
    else:
        insert new House row (new surrogate id)

for each activeRow whose key is NOT present in apiItems:
    setEndDate(now)
```

Notes:

- Ended rows are never touched. A re-listed property always becomes a new row because its `availableFrom` differs.
- The `setEndDate(null)` line in the current code goes away. Re-activation is no longer a thing.
- API items missing `availableFrom` are dropped per the policy above (log warning, skip).
- The "set area-level address only for new houses" behaviour is preserved.

### 4. Deadline-based ending

A separate sweep at the start of `syncHouseList`:

```
markPastDeadlineEnded():
    update House h
       set h.endDate = now
     where h.endDate is null
       and h.applicationDeadline is not null
       and h.applicationDeadline < now
```

Implemented as a single JPQL bulk update (or a small native query) for efficiency. Runs once per `syncHouseList` tick, before the API fetch, so the active set used by the reconcile is already pruned. Houses past their deadline therefore drop out of both `syncHouseList`'s active map and `syncHouseDetails`'s refresh queue immediately.

We deliberately do **not** check the deadline inside `syncHouseDetails` — keeping the state transition in one place is clearer than scattering it across both jobs.

### 5. Adaptive detail-fetch cadence

Replace `@Scheduled(fixedDelay=...)` with a dynamic-delay scheduler.

Approach: a `@PostConstruct` method on `HouseSyncService` schedules itself with Spring's `TaskScheduler`. Each invocation:
1. fetches the next stale active house (`findHousesNeedingDetailRefresh` with `PageRequest.of(0, 1)`),
2. processes it (or returns immediately if none),
3. computes the next delay,
4. re-schedules itself.

Delay calculation:

```
activeCount = houseRepository.countByEndDateIsNull()
if activeCount == 0:
    delay = MAX_IDLE_DELAY  // e.g. 30 minutes
else:
    delay = clamp(
        refreshWindow / activeCount,
        MIN_DELAY,   // e.g. 30 seconds — API politeness floor
        MAX_DELAY    // e.g. 30 minutes
    )
```

Where `refreshWindow = detailRefreshIntervalHours`. With 50 active houses and a 12h window, that's one detail call every ~14 min. With 5 active houses, every ~2.5h. With 0, every 30 min (just polling for new active houses).

The bounds are config-driven (new env vars `DETAIL_FETCH_MIN_DELAY_SECONDS`, `DETAIL_FETCH_MAX_DELAY_SECONDS`, `DETAIL_FETCH_IDLE_DELAY_SECONDS`) but have sensible defaults so the operator usually doesn't touch them.

`detailSyncBatchSize` is removed: each tick handles exactly one house. Latency to refresh any individual house is `delay × position-in-queue`, which under the formula above is bounded by `refreshWindow`.

### 6. Env-var and config cleanup

Remove:

- `DETAIL_API_CALL_INTERVAL_SECONDS` (replaced by adaptive cadence)
- `DETAIL_SYNC_BATCH_SIZE` (each tick = one house)
- Any matching `appconfig.*` keys in `application.yml`
- Any matching getters in `AppConfig`
- The corresponding rows in `CLAUDE.md`'s env-var table

Add:

- `DETAIL_FETCH_MIN_DELAY_SECONDS` (default 30)
- `DETAIL_FETCH_MAX_DELAY_SECONDS` (default 1800)
- `DETAIL_FETCH_IDLE_DELAY_SECONDS` (default 1800)

Audit step in the plan: grep for any other env vars / `AppConfig` fields that became unreachable after the legacy notification removal but were missed in that pass.

## Data flow (end state)

```
syncHouseList (every listApiCacheDurationHours):
    1. markPastDeadlineEnded()
    2. fetch API list
    3. load active rows
    4. reconcile by (id, availableFrom): insert / update / end-date
    5. save

syncHouseDetails (adaptive, single-house tick):
    1. find next stale active house
    2. fetch detail
    3. apply detail (queue points sticky on null)
    4. save with new lastDetailFetchedAt
    5. compute next delay from active-count, re-schedule
```

## Testing

Unit tests (Mockito + an embedded DB for repository tests where the JPQL matters):

- **Sticky queue points:** non-null → write; null with prior non-null → no write; null with prior null → no write.
- **Reconcile matching:** new `(id, availableFrom)` → insert; existing match → update; missing from API → end-date; same `id` with different `availableFrom` → new row, old row untouched.
- **Past-deadline sweep:** rows with `applicationDeadline < now` and `endDate is null` get ended; rows with no deadline are unaffected; already-ended rows are unaffected.
- **Adaptive delay:** zero active → idle delay; small active count → max-clamped; large active count → min-clamped; in-between → expected division.

## Migration / rollout notes

- Liquibase changeset adds the surrogate PK and unique constraint in one transaction.
- If the existing table has duplicate `(id, available_from)` pairs (it shouldn't, since `id` was previously unique), the migration fails fast — that's the desired behaviour.
- No application downtime required; the new code paths only activate after the migration runs.
